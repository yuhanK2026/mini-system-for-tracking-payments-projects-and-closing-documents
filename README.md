# Payment Tracking Dashboard System

## Описание

Мини-система учета оплат, проектов и закрывающих документов для digital-агентства.

Показывает связь: **проект / юрлицо → оплаты → этапы работ → статус закрывающих документов**

## Сущности и связи

### Сущности

| Сущность | Описание | Поля |
|----------|----------|------|
| **Client** | Юридическое лицо (клиент) | id, name, inn, ogrn |
| **Project** | Проект | id, name, client_id, status |
| **Payment** | Оплата | id, project_id, client_id, payment_date, amount, payment_purpose, service_stage |
| **Act** | Закрывающий документ | id, payment_id, is_sent, sent_at, is_signed, signed_at, manager_comment |

### Связи

```
Client (1) ──────< Project (M)
Project (1) ─────< Payment (M)
Client (1) ──────< Payment (M)
Payment (1) ─────> Act (1)
```

- Клиент может иметь много проектов
- Проект принадлежит одному клиенту
- Проект может иметь много оплат
- Оплата принадлежит проекту и клиенту
- Каждая оплата может иметь один акт

## Бизнес-логика

### Статусы актов

| Статус | Условие |
|--------|---------|
| **Не отправлен** | Акт не отправлен и не подписан |
| **Ожидает подписи** | Акт отправлен, не подписан, отправлен ≤30 дней назад |
| **Закрыт** | Акт отправлен и подписан |
| **Требует внимания** | Акт отправлен >30 дней назад, но не подписан ИЛИ оплате >60 дней, акт не отправлен |

### Расчет итогов

- **Общая сумма**: сумма всех оплат
- **Сумма закрытых актов**: сумма оплат с подписанными актами
- **Сумма незакрытых актов**: сумма оплат без подписанных актов
- **Без отправленного акта**: количество оплат без отправленного акта
- **Отправлен, не подписан**: количество актов в статусе "Ожидает подписи"

## API Endpoints

### Dashboard

```
GET /api/dashboard
```

Параметры:
- `project_id` - фильтр по проекту
- `client_id` - фильтр по клиенту
- `act_status` - фильтр по статусу акта (not_sent, awaiting_signature, closed, attention_required)
- `start_date` - начало периода
- `end_date` - конец периода
- `search` - поиск по назначению платежа или названию клиента

### Управление актами

```
POST /api/acts         - Создать акт
PATCH /api/acts/{id}   - Обновить статус акта
```

## Технологический стек

- **Backend**: Laravel 11
- **Frontend**: Vue.js 3 + Vite
- **Database**: SQLite (MySQL/PostgreSQL)
- **API**: REST

## Установка и запуск

```bash
# Установка зависимостей
composer install
npm install

# Настройка БД
php artisan migrate:fresh --seed

# Сборка фронтенда
npm run build

# Запуск сервера
php artisan serve
```

Открыть: http://localhost:8000

## Допущения и решения

1. **Связь Payment → Client**: Дублирование client_id в таблице payments сделано для удобства запросов и независимости от Project
2. **Статусы актов**: Реализованы в виде мутатора в модели Act для автоматического пересчета
3. **Фильтрация**: Реализована на стороне backend для эффективной работы с большими данными
4. **UI**: Простой, без сложного дизайна - главное логика
